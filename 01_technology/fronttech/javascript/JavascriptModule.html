# Javascript模块化

# 1. 模块之前

Javascript是为在web浏览器中使用而设计的。为了使用JavaScript，必须在html中添加<script>标签，然后代码被下载和执行。早期JavaScript使用的相当少，如果想要使用jQuery库，你必须引入<script>标签加载它，然后追加另一个<script>标签加载自己的代码。jQuery库负责将自己绑定到全局上下文中（window.$），然后你的代码就可以发现并使用它。
这种方式至今仍然起作用，但是它有缺点：

  • <script>标签的顺序问题。必须首先定义全局变量，否则其他依赖全局变量的代码就会报错。
  • 没有办法排除库中未使用到的代码，尤其是经过最小化或混淆过的。如果想使用库或框架中的一部分，必须加载全部。
  • 如果两个库定义了同一个名字的全局变量，没有覆盖的情况下你不能同时使用这两个库。
  • 如果你把单个JavaScript拆分成多个文件，必须使用绑定到全局上下文的技术来共享状态或函数。

# 2. 模块的产生

Javascript刚出现的时候程序都非常小，大都是做一些增加web页面交互的独立脚本任务，故大的程序基本不需要。经过这么多年的发展，我们现在已经有大量javascript组成的完整程序运行在浏览器中，同时在不同的环境中被使用，如nodejs。

Node.js铺平了在web外使用JavaScript的道路。它提供了一个可以在计算机上执行JavaScript代码的运行环境，而不依赖于浏览器。伴随着Node.js，使JavaScript代码模块化的需求更加强烈。没有html标签，当需要库时如何加载呢？Node.js需要一种方式来声明对其他JavaScript代码的依赖，并不是因为模块化使事情更容易，而是因为JavaScript不在运行于web页面上下文中。

随着javascript被越来越多的应用所使用，管理和维护代码成为一件困难的事情。除此之外，在同一个项目中使用多个javascript文件会引起命名冲突，并产生巨大风险。
因此模块的概念应运而生，用于解决两个重要问题：

  • 模块必须有一个封闭的空间，所有向外提供的信息必须是明确清晰的
  • 模块必须将大的应用程序分割许多单独的小组件，使开发和维护应用程序更容易

# 3. 模块化简史

# 3.1. IIFE

IIFE(Immediately Invoked Function Expression)


这是模块化代码最原始的技术。这个时候javascript中还没有类的概念。IIFE提供了在封闭空间创建变量、函数供内部使用的自由，不用担心应用程序中发生冲突的问题。

IIFE表达式类似这样：

(function () {
  // Statements
})();

var myGlobalVariable = 'Hello, I am a global variable :)';

    return publicText;
})();

# 3.2. CommonJS

CommonJS是第一个真正流行起来的JavaScript模块系统。Node.js进行了采用，随着Node.js的崛起，也带来了CommonJS的普遍性。CommonJS项目不仅仅是模块规范，还是创建标准库和代码分发平台的整体设想，这一设想的成功今天看来由于NPM的巨大普及。

CommonJS规范是由Mozilla工程师Kevin Dangoor于2009年1月提出。它主要为Javascript环境建立了模块生态约定，这里不包括浏览器环境。实际上它最初的名字叫ServerJS，目标是服务端生态系统。2009年8月被命名为CommonJS，以表达其更广泛的应用。

自成立以来，CommonJS规范一直被NodeJS采用。目前NodeJS也已经支持ES6 Module规范。CommonJS的识别特征是require()和module.exports()函数。每个CJS模块都有一个module的全局对象，附着一个exports属性。任务分配给module.exports的内容，对其他模块来说都是可用的。

优点：

  • 简单且易用
  • 内建依赖管理，模块以合适的顺序加载
  • require()可以在程序中任何需要其他模块的地方使用
  • 支持循环依赖

缺点：

  • 同步加载模块导致加载速度缓慢，对于浏览器端的应用程序并不合适
  • 一个文件一个模块
  • 浏览器不能根据CommonJS标准识别模块，需要额外的模块加载器库或者传输器
  • 不支持构造函数，但是NodeJS支持
  • 静态代码分析器很难分析

实现CommonJS规范的库有：

  • NodeJS (Server side)
  • webpack (Client side)
  • Browserify (Client side)

声明一个模块

function _add(a, b) {
    return a + b;
}
function _subtract(a, b) {
    return a - b;
}
module.exports = {
    add: _add,
    subtract: _subtract,
};

# 3.3. AMD

AMD (Asynchronous Module Definition)

AMD诞生于一群对CommonJS提倡方向不满的开发者。实际上AMD是早期从CommonJS中分离出来的，主要区别就是前者支持异步模块加载，同样是一个javascript规范。

优点：

  • 异步加载，良好的启动时间
  • 支持并兼容require()和module.exports()，类似于CommonJS
  • 内建依赖管理
  • 一个模块可以包含多个文件
  • 支持构造函数
  • 支持循环依赖
  • 插件支持（自定义加载步骤）

缺点：

  • 语法上比CommonJS略微更复杂
  • 静态代码分析器很难分析
  • 浏览器不能根据AMD标准识别模块，需要额外的模块加载器库或者传输器

创建模块：

define("myAmdModule", {
    add: function (a, b) {
        return a + b;
    },
    subtract: function (a, b) {
        return a - b;
    },
});

# 3.4. UMD

UMD (Universal Module Definition)

UMD是一个技术集，创建模块时可以导入IIFE、CommonJS或AMD模块。意味着开发人员可以导入第三方模块，无论当前使用那种模块规范。

(function (myUmdModule) {
    if (typeof define === 'function' && define.amd) {
        // Export with AMD
        define("myUmdModule", myUmdModule);
    } else if (typeof module === 'object' && module.exports) {
        // Export with CommonJS
        module.exports = myUmdModule;
    } else {
        // Export with browser global
        window.myUmdModule = myUmdModule;
    }
}({
    add: function (a, b) {
        return a + b;
    },
    subtract: function (a, b) {
        return a - b;
    },
}));

# 3.5. ES Modules

ES Modules又称为ECMAScript Modules、ES6 Modules或ES2015 Modules。ES Modules是模块生态美好未来的解决方案，通过标准的模块声明语法，前后端都可以工作，从JavaScript语言层次上进行支持。但是ES Modules出现的较晚，很多使用者和库还没有完全迁移至此规范，尚需时间。
